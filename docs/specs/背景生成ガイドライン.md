# 背景生成ガイドライン

最終更新: 2026年5月30日

## 目的

genDeckの背景は、単体で美しい画像ではなく、カスタムGPTが作るdeck JSONとPPTX描画に合う「スライドの土台」として作る。

背景生成では、顧客ごとの世界観を反映しつつ、文字、表、写真、QR、申込情報が上に載ることを最優先する。背景は事前に合格を取り、生成時は背景プールから選ぶ。

## 基本原則

- 16:9横長で作る
- 中央または主要テキストエリアに十分な余白を残す
- 装飾は端、角、外周に寄せる
- 読める文字、ロゴ、ブランド名、日付、数字を背景画像に入れない
- 背景単体で完成させすぎず、上に日本語テキストが載って完成する密度にする
- 顧客の世界観は、色だけでなく素材、質感、モチーフ、余白の取り方で表現する
- 1枚だけで判断せず、実際に `build_deck.py` でPPTX化した見え方で確認する

## JSON typeと背景の対応

カスタムGPTが出す `slides[].type` は、背景選択にも影響する。背景を作るときは、どのtypeで使うかを先に決める。

| JSON type | 既定背景 | 背景の役割 | 背景要件 |
| --- | --- | --- | --- |
| `cover` | `paper_stage` | 世界観を最初に伝える | 密度はやや高めでよいが、タイトル領域は広く空ける |
| `statement` | `paper_stage` | 章扉、問い、強い一文 | 中央または左右どちらかに大きな余白。文字の邪魔をしない |
| `process` | `open_center` | 流れ、ステップ | 3カラムや横並び要素が載るため、中央横方向を広く空ける |
| `choice` | `open_center` | 選択肢、比較 | 2〜4カラムを想定し、背景のコントラストを抑える |
| `content` | `open_center` | 写真、資料画像、ワークシート画像 | 画像カードや資料画像が載る前提で静かな背景にする。画像なし説明には使わない |
| `offer` | `open_center` | 商品、特典、価格 | 価格や特典が目立つよう、中央を整理する |
| `cta` | `postcard_cta` | 申込、QR、最終案内 | QRやURL、申込文言が読めるよう、下部/中央の余白を守る |
| `multi_image` | `photo_frame` | 写真複数枚 | 写真カードが載るため、背景装飾は控えめにする |
| `before_after` | `photo_frame` | 比較写真 | 左右2枚の写真が載るため、左右中央を空ける |
| `three_column` | `open_center` | 3列説明 | カラムの読みやすさ優先。柄や強い濃淡を避ける |
| `person` | `photo_frame` | 人物紹介 | 人物写真と本文の両方が載るため、端に世界観を寄せる |

## 背景スタイル

現在の実装では、互換のため旧type背景も読める。ただし、本番用に新規作成する背景はstyle形式だけにする。

優先して使うstyle:

- `paper_stage`: 表紙、章扉、強い一文向け。紙面や舞台感があり、世界観を出しやすい
- `open_center`: 本文、比較、ステップ、オファー向け。中央の可読性を最優先する
- `postcard_cta`: CTA向け。申込、QR、URLが自然に乗る
- `photo_frame`: 写真、人物、before/after向け。画像カードを置く前提

保存形式:

```text
backgrounds/{style}/{color}_normal.png
backgrounds/{style}/{color}_reverse.png
```

旧形式はlegacy fallback専用。新規背景では使わない:

```text
backgrounds/{color}/{type}.png
backgrounds/{color}/{type}_02.png
```

新規で作る背景は、必ずstyle形式を使う。旧形式は既存デッキとの互換確認や一時的なfallbackに限定する。

## 顧客別プロファイル

色、トーン、顧客固有の世界観、NG表現、合格した背景プロンプトは、顧客情報を含む可能性があるため、この汎用ガイドラインには書かない。Git管理しないローカルの顧客別プロファイルにまとめる。

推奨ローカル保存先:

```text
local/customers/{customer_id}/背景スタイル.md
```

背景生成前に、顧客ごとのスタイル定義を短く書く。`local/` 配下はGit管理しない。

最低限決めること:

- 世界観の名前
- 色とトーン
- 主役にする素材
- 脇役にするモチーフ
- 避ける印象
- 文字を載せる余白の取り方

```text
Primary request:

Composition:
16:9 landscape.
Reserve a large clean area for Japanese slide text.
Place decorative objects mostly along the edges and corners.

Output:
One polished full-bleed background image, high detail, clean enough for overlaying Japanese text.
```

## 顧客別プロンプト例

顧客別の具体プロンプトは、Git管理しないローカルの顧客別プロファイルに保存する。この汎用ガイドラインには、テンプレートとチェック観点だけを残す。

## QAチェックリスト

背景を採用する前に、必ず確認する。

- 16:9でトリミングしても主役が崩れない
- 中央または想定テキスト位置に十分な余白がある
- 日本語タイトルを置いて読める
- 本文を置いて読める
- QRや価格など重要情報を置く場所がある
- 読める英字、数字、ロゴ、謎のラベルがない
- 背景の主張が強すぎない
- 顧客のNG印象に寄っていない
- 同じデッキ内で他背景とトーンが揃う
- `build_deck.py` で実際にPPTX化して見た目を確認した

## 運用フロー

1. 顧客の参考画像やNG感を確認する
2. このガイドラインに沿って、顧客別スタイル定義を作る
3. imagegenで候補を1〜3枚作る
4. 顧客またはじゅじゅが合格/不合格を判断する
5. 合格背景を、用途に合わせてローカルの `backgrounds/` に登録する
6. カスタムGPTが作ったdeck JSONでPPTX生成する
7. 実スライド上で可読性と雰囲気を確認する
8. 必要なら背景、JSON type、スライド文量のどれを直すか切り分ける
9. 採用背景は顧客IDつきzipとしてまとめ、R2などのprivate storageへ置く。Gitには含めない

## 不合格時の切り分け

背景が合わないときは、原因を分けて考える。

| 症状 | 直す場所 |
| --- | --- |
| 文字が読みにくい | 背景の余白、明度、コントラスト |
| ページがごちゃつく | JSON側の情報量、スライドtype、背景の密度 |
| 世界観が違う | 顧客別スタイル定義、主役素材、NG指定 |
| 意図より装飾的・軽い印象になる | モチーフの主従、色、質感 |
| 申込ページが弱い | `cta`背景、文言、QR/価格の配置 |
| 写真が浮く | `photo_frame`背景、写真枠、背景の装飾量 |

## 保存メモ

生成した候補は、採用前に不用意に既存背景へ上書きしない。

推奨:

```text
backgrounds/{style}/{color}_normal.png
backgrounds/{style}/{color}_normal_02.png
```

採用した背景は、どの顧客、どのtype、どの用途で合格したかをメモしておく。
